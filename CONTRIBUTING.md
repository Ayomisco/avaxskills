# Contributing to Avalanche Skills

Thank you for contributing. Every skill you improve helps developers build better on Avalanche.

---

## Skill Format (V2 Spec)

Every `SKILL.md` must follow this structure. CI validates on every PR.

### YAML Frontmatter (Required)

```yaml
---
name: "skill-name"              # kebab-case, matches folder name
version: "1.0.0"                # semver
spec: "agentskills@1.0"         # compatibility marker — do not change
license: "Apache-2.0"           # required
tier: 1                         # 0-6, see tiers below
description: "One sentence. Include the trigger phrases an agent would use."
trigger: |
  Use when: [specific triggers].
  Do NOT use for: [anti-triggers].
last_updated: "2026-05-01"
avalanche_networks: [fuji, mainnet]   # omit if not network-specific
related_skills:
  - skill-name-1
  - skill-name-2
---
```

### Required Sections

| Section | Purpose | Token budget |
|---|---|---|
| Overview | 2–4 sentences. What does this skill enable? | < 100 tokens |
| When to fetch | Explicit triggers + anti-triggers | < 80 tokens |
| Core Workflow | Numbered steps. Exact commands. No hand-waving. | < 600 tokens |
| Network config | Chain ID, RPC, explorer (for network skills) | < 100 tokens |
| Key concepts | ≤5 Avalanche-specific definitions | < 150 tokens |
| Common errors | Table: Error / Cause / Fix — top 3–5 only | < 150 tokens |
| Next skills | Which skills to fetch next | < 50 tokens |

**Hard cap: SKILL.md must be under 500 lines.** CI rejects PRs that exceed this.
- Heavy content → `references/`
- Guardrails → `rules/`
- Runnable code → `scripts/`

## Folder Structure (Per Skill)

```
skills/{skill-name}/
├── SKILL.md          # Required. Core workflow. < 500 lines.
├── references/       # Optional. Deep docs. Loaded on demand.
│   └── *.md
├── rules/            # Optional. Hard guardrails agents MUST follow.
│   └── *.md
├── scripts/          # Optional. Runnable bash/ts helpers.
│   └── *.sh / *.ts
└── {skill-name}.zip  # Auto-generated by CI. Do not commit manually.
```

### The rules/ directory

`rules/` is separate from `references/`. Rules are **opinionated constraints** that agents must follow — not optional reading.

Good rules:
- "Never set gas limit below 8M on Subnet-EVM"
- "Always verify the Teleporter address before sending cross-chain messages"
- "Do not use tx.origin for access control on Avalanche"

Bad rules (these belong in references/):
- "Here is how gas works on Avalanche" ← explanation, not a guardrail
- "You can use Hardhat or Foundry" ← option, not a constraint

## Submitting a New Skill

1. Fork the repo
2. Create `skills/{your-skill-name}/SKILL.md`
3. Run validation: `npm run validate`
4. Open a PR — CI runs automatically
5. Skills are reviewed for: accuracy, specificity, token efficiency, and actual usefulness to AI agents

---

## Updating an Existing Skill

- Bump `version` in the frontmatter
- Update `last_updated`
- If you change a command or address, verify it works on Fuji first

---

## What Makes a Good Skill

**Good:**
- Exact commands with real flags, real addresses, real chain IDs
- Common errors an AI agent would actually encounter
- Specific to Avalanche — not generic Ethereum content
- Triggers that are precise enough for an agent to know when to fetch it

**Bad:**
- Generic "here is Solidity" content
- Commands without context (just `npm install` with no config)
- Aspirational descriptions of what the skill *could* cover
- Duplicate content that belongs in `references/`
